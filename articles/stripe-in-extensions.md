Stripe in Extensions

Why Stripe is the best option for extension payments today. Chrome Web Store payments were deprecated in 2020 and Google never replaced them. Stripe gives you full control, supports subscriptions and one-time payments, and works everywhere. PayPal and Paddle exist but Stripe has the best developer experience and the broadest feature set.

The payment flow works like this. User clicks an upgrade button in the extension popup or options page. The extension opens a Stripe Checkout URL in a new tab. The user pays on a Stripe-hosted page so you never touch card data. Stripe fires a webhook to your server. Your server updates the user's payment status in your database. The extension checks payment status on next load or via a polling interval and unlocks premium features. This flow is simple, secure, and PCI compliant by default.

Setting up Stripe Checkout requires a few steps. Create products and prices in the Stripe Dashboard. Use price IDs in your code, never hardcode dollar amounts. For subscriptions, create a recurring price. For lifetime deals, create a one-time price. Generate Checkout Session URLs server-side using the Stripe SDK. Pass the customer email so Stripe can associate the purchase. Set success and cancel URLs that redirect back to your extension or landing page. Never handle raw card data in your extension or your server.

Connecting the extension to Stripe identity takes one of three approaches. First, use Chrome identity API to get the user's Google email, then match that email to a Stripe customer. Simple but ties the user to their Google account. Second, build a custom account system with email and password, then link to Stripe customer. More work but more flexible. Third, use license keys generated after payment with no account needed. Best for simple extensions that just need a gate. Each approach works, pick based on how much infrastructure you want to maintain.

Webhook handling is where your server does the real work. The events you must handle are checkout.session.completed which fires when a payment succeeds, customer.subscription.updated which fires when a subscription changes status, customer.subscription.deleted which fires when someone cancels, and invoice.payment_failed which fires when a renewal charge fails. For each event, describe what your server should do. Verify the webhook signature. Extract the customer email or ID. Update your database. Optionally notify the user. Webhooks are your source of truth, not the extension checking status.

Testing thoroughly before going live prevents real headaches. Use Stripe test mode with test card numbers. Simulate successful payments, declined cards, and insufficient funds. Test subscription renewals by setting short billing intervals. Test cancellation flow and verify the extension properly reverts to free. Test what happens when your server is down and the extension cannot check status. Users should get a grace period, not an instant lockout.

Common pitfalls trip up most implementations. Status caching is critical because you cannot call your server on every single extension action, so cache the payment status locally with a reasonable TTL like 1 to 6 hours. Set up Stripe Customer Portal so users can manage their own subscriptions without emailing you. Handle currency and tax properly, Stripe Tax can automate most of this. Do not forget to handle the case where a user pays on one device and opens the extension on another. These details matter more than you think.

The Zovo Pro payment flow at zovo.one is built on this exact Stripe pattern. We handle subscriptions across all 17 extensions with a single backend and shared customer database. The same code that processes webhooks for one extension works for all of them. When a user upgrades, they get access to every extension we make. This approach scales and keeps maintenance manageable. If you are building monetization into an extension, Stripe plus webhooks plus a simple status check is the path that works.
